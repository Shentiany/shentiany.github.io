  • 避免使用容易引起混淆的名称

建议4：在代码中适当添加注释

  • 注释与代码重复。注释应该是用来解释代码的功能、原因以及想法的，而不是对代码本身的解释。
  • 利用注释语法快速删除代码。对于不再需要的代码，应该将其删除，而不是将其注释掉。即使你担心以后还会用到，版本控制工具可以帮你轻松找回被删除的代码。

建议5： 通过适当添加空行使代码布局更为优雅、合理

  • 在函数定义或者类定义之间空两行。在类定义与第一个方法之间，或者需要进行语义分割的地方空一行。
  • 尽量保持上下文语义的易理解性。当一个函数需要调用另一个函数的时候，最好调用者在上，被调用者在下。
  • 避免过长的代码。
  • 不要为了保持水平对齐而使用多余的空格。
  • 二元运算符（赋值（=），比较（==, <, >, !=, <=, >=, in, not in, is, is not ）、布尔运算（and, or, not））的左右两边应该要有空格。
  • 逗号和分号前不要使用空格。
  • 函数名的左括号之间、序列索引操作时系列名和[]之间不需要空格，函数的默认参数两侧不需要空格。
  • 强调前面的操作符的时候使用空格，如-2 - 5（在-2和5之间的减号前后需要添加空格）、bb + ac(在加号前后需要添加空格)

建议6：编写函数的4个原则

  • 原则1：函数设计要尽量短小，嵌套层次不宜过深。
  • 原则2：函数声明应该做到合理、简单、易于使用。
  • 原则3：函数参数设计应该考虑向下兼容。
  • 原则4：一个函数只做一件事，尽量保证函数语句粒度的一致性。

建议7：将变量集中到一个文件

如果代码对应的模块名为const，使用时只需要import const，便可以直接定于常量了。

建议10： 充分利用Lazy evaluation的特性

  • 如果对于or条件表达式应该将值为真可能性较高的变量写在or的前面，而and则应该推后。

建议11：理解枚举代替实现的缺陷

枚举的各种替代实现方式：

  • 使用类属性

  • 借助函数

  • 使用collection.namedtuple

建议12：不推荐使用type来进行类型检查

  • 基于内建类型扩展的用户自定义类型，type函数并不能准确返回结果
  • 在古典类中，所有类的实例的type值都相等

建议13：尽量转换为浮点类型后再做除法

  • 当涉及除法运算的时候尽量先将操作数转换为浮点类型再做运算(整数除法问题在Python3中已经不存在)

建议14：警惕eval（)的安全漏洞

  • 如果使用对象不是信任源，应该尽量避免使用eval，在需要使用eval的地方可用安全性更好的ast.literal_eval代替。

建议15：使用enumerate（）获取序列迭代的索引和值

建议16：分清==与is的适用场景

  • == 是比较两个对象的内容是否相等，不管两者在内存中的引用地址是否一样，不管两者的类型
  • is 比较的是两者的地址是否相同

建议17：import的使用

  • 优先使用 import math 形式
  • 有节制的使用from math import sin 形式
  • 避免使用from math import * 形式

建议18：连接字符串应优先使用join而不是+

  • 使用join()方法和使用+操作符来连接字符串时，join()方法的效率要明显高于+操作符

建议19：慎用可变参数

  • 在函数定义时可以使用* args和** kwargs实现可变长度的参数列表
  • 可变参数在下列情况下使用
    • 为函数添加一个装饰器
    • 如果参数的数目不确定
    • 用来实现函数的多态或者子啊继承情况下子类需要调用父类的某些方法的时候
